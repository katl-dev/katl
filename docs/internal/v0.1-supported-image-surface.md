# v0.1 Supported In-Image Surface

Status: accepted support boundary for v0.1 image trimming.

This document defines what KatlOS v0.1 intentionally exposes from the installer
image, runtime rootfs, and Kubernetes payload sysext. The purpose is support
minimization: remove or fence off visible behavior that implies a product
contract Katl does not intend to maintain.

The installed runtime keeps broad kernel hardware support where Fedora packaging
permits it. The ephemeral installer retains the boot, storage, filesystem,
wired-network, input, and virtual-machine module families needed to install that
runtime, but does not select runtime-only GPU, wireless, media, or sound support.
Transitive module dependencies may still be present. See
`docs/internal/v0.1-kernel-firmware-support-policy.md`.

## Rules

- A command, service, path, or package being present is not enough to make it a
  supported operator interface.
- Supported surfaces must have a Katl-owned workflow, validation path, and VM or
  unit gate.
- Internal implementation details may exist in the image, but CI should prevent
  users from depending on them by denying stable entrypoints where practical.
- Debug surfaces must be opt-in and clearly tied to VM, installer recovery, or
  break-glass flows.
- Package-manager and build-tool surfaces are unsupported in installed runtime
  roots. Katl updates through Katl artifacts, systemd-sysupdate, sysext/confext,
  and explicit `katlc` operations.
- Kubernetes and node app payloads are not installer/rootfs userland. They are
  fetched and staged from HTTPS bundles and selected by generations.

## Installer Image

Supported v0.1 installer commands:

| Command | Surface | Notes |
| --- | --- | --- |
| `katlos-install --boot` | supported | systemd-started installer entrypoint. |
| `katlos-install` subcommands needed by installer state tests | supported | only documented installer workflows are stable. |
| `katl-console` | supported internal | read-only `tty1` operator dashboard; not a configuration API. |
| `systemctl`, `journalctl`, `networkctl`, `resolvectl`, `ip` | debug/break-glass | allowed for installer diagnosis, not a configuration API. |
| `ssh`/`sshd` | conditional support | only when installer config enables `/etc/katl/installer-ssh.enabled`; key-only root access. |
| `curl` | internal installer fetch helper | not a supported general download tool. |
| `bash` | debug/break-glass and script interpreter | not a supported extension mechanism. |
| installer console autologin | debug/break-glass | acceptable for v0.1 VM and recovery loops, not a remote access model. |

Supported installer services and units:

| Unit | Surface | Notes |
| --- | --- | --- |
| `katl-installer.target` | supported | dedicated installer boot target; keeps the initrd environment active without misreporting the installer as starting. |
| `katlos-install.service` | supported | boot-time installer operation. |
| `katl-console.service` | supported | owns VGA `tty1`; `tty2` and SSH remain separate recovery surfaces, while serial carries logs when available. |
| `katl-input-apply.service` | supported | applies installer handoff input. |
| `katl-installer-smoke.service` | VM/test-only | smoke signal, not a production install contract. |
| `systemd-networkd.service` and `systemd-resolved.service` | supported | installer networking and DNS. |
| `sshd.service` | conditional support | gated by `/etc/katl/installer-ssh.enabled`. |

Unsupported installer surface:

- Package-manager workflows: `dnf`, `dnf5`, `rpm`, `yum`, or repository files
  as an operator interface.
- Build/development tools: `gcc`, `go`, `make`, `mkosi`, compilers, linkers.
- Kubernetes administration: `kubeadm`, `kubelet`, `kubectl`, `crictl`,
  `helm`, CNI CLIs, GitOps tools.
- Generic systemd containers or app sandboxes: `systemd-nspawn`,
  `machinectl`, portable service management.
- Arbitrary shell hooks, arbitrary unit injection, or package installation from
  installer input.
- Long-lived installer SSH without explicit config input and key-only auth.

Installer trimming Beads should move unsupported but useful developer tools into
dev/debug profiles, VM-only fixtures, or host-side tooling. If a dependency
pulls in an unsupported command, CI should either deny the command path or record
why it cannot be removed yet.

## Runtime Rootfs

ADR-005 defines the v0.1 container-runtime boundary: `containerd`, the selected
OCI runtime, Katl-owned service ordering, and `/var/lib/containerd` state remain
base runtime responsibilities and are kept current with the latest supported
KatlOS base/runtime validation gates. This does not make KatlOS a user-facing
container platform or a production CNI manager.

Supported v0.1 runtime commands:

| Command | Surface | Notes |
| --- | --- | --- |
| `katlc` | supported | node-local state/config/operation command and agent CLI. |
| `katl-console` | supported internal | renders node, network, generation, health, and journal state on VGA `tty1`. |
| `katl-boot-health`, `katl-runtime-status`, `katl-generation-activate`, `katl-ssh-host-keys` | internal supported | stable as Katl unit entrypoints, not operator CLIs. |
| `systemctl`, `journalctl`, `networkctl`, `resolvectl`, `bootctl`, `importctl` | break-glass/debug | allowed for inspection and recovery, not declarative config. |
| `ssh`/`sshd` | supported node access | used for operator recovery and VM tests; configured through Katl-owned input/state. |
| `containerd`, `crun` | internal Kubernetes runtime | managed by Katl/systemd, not a user container platform. |
| `conntrack`, `iptables-nft`, `nft`, `ipset`, `ipvsadm` | internal Kubernetes prerequisites | present for kube-proxy/CNI diagnostics and runtime support. |
| CNI binaries under `/usr/libexec/cni` | internal Kubernetes prerequisites | not a supported CNI management interface. Mutable CNI installers use `/var/lib/katl/kubernetes/cni`; conventional `/opt/cni/bin` and `/etc/cni/net.d` paths resolve there. |
| `kmod`, `modprobe`, `lsmod`, `ip`, `mount`, `findmnt`, `blkid` | break-glass/debug | available for diagnosis and VM evidence. |

Supported runtime services and targets:

| Unit | Surface | Notes |
| --- | --- | --- |
| `katlc-agent.service` | supported | node management endpoint. |
| `katl-console.service` | supported | read-only VGA status and journal dashboard. |
| `katl-generation-activate.service` | supported internal | generation activation unit. |
| `katl-boot-complete.target`, `katl-boot-health.service`, `katl-boot-deadman.*` | supported | boot health contract. |
| `katl-runtime-handoff-status.service` | supported | runtime status and handoff evidence. |
| `katl-ssh-host-keys.service` | supported internal | validates persistent SSH identity and atomically repairs missing or invalid keys before sshd starts. |
| `katl-kubeadm-ready.target` | supported internal | readiness gate for kubeadm operations. |
| `containerd.service` | supported internal | Kubernetes/container runtime dependency. |
| `kubelet.service` drop-in | supported when Kubernetes sysext is selected | rootfs owns ordering/mount integration only. |
| `var.mount`, `etc-kubernetes.mount` | supported | persistent state and kubeadm projection. |
| `systemd-networkd`, `systemd-resolved`, `systemd-timesyncd`, `dbus-broker` | supported platform services | configured through Katl-owned domains. |
| `katl-vmtest-agent.service`, `katl-vmtest-debug-shell.service` | VM/test-only | gated by kernel command line; never a production API. |

Unsupported runtime surface:

- Package-manager workflows: `dnf`, `dnf5`, `rpm`, `yum`, repository
  configuration, or mutable package installation.
- Build/development tools: `gcc`, `go`, `make`, `mkosi`, compiler toolchains.
- Generic download/admin tooling as product API: `curl`, `wget`, editors,
  general-purpose scripting libraries, ad hoc shell automation.
- Kubernetes CLIs in the runtime root: `kubeadm`, `kubelet`, `kubectl`, `crictl`
  belong to the fetched Kubernetes payload sysext.
- Helm, Flux, Cilium CLI, Rook tooling, CoreDNS management, or Kubernetes add-on
  lifecycle.
- `systemd-nspawn`, `machinectl`, and generic systemd-container workflows as
  operator features. If `systemd-container` is present for `importctl` or
  sysupdate implementation, these entrypoints remain unsupported.
- `dracut` as an operator tool. It is build-only plumbing and is removed after
  the runtime UKI has been assembled.
- `procps-ng` and `psmisc` as stable scripting APIs. Tools such as `ps`, `top`,
  `killall`, `pstree`, `pkill`, or `fuser` may be used for break-glass
  diagnosis only.
- Arbitrary systemd unit deployment, arbitrary `/etc` mutation, or unmanaged
  CNI/kubeadm file ownership.

Runtime trimming Beads should preserve the supported Katl and Kubernetes
prerequisite surface while removing unsupported package/build/download tools or
fencing them behind dev/debug profiles. If a command remains because it is a
hard dependency, tests should document that it is present but unsupported.

## Kubernetes Payload Sysext

Supported v0.1 sysext commands:

| Command | Surface | Notes |
| --- | --- | --- |
| `kubeadm` | supported through Katl/kubeadm operations | not a general host admin API. |
| `kubelet` | supported as systemd-managed Kubernetes node agent | activation is generation-gated. |
| `kubectl` | operator/test helper when explicitly used by workflows | not a replacement for Katl state operations. |
| `crictl`, `ethtool`, `socat` | internal/bootstrap diagnostics | included for kubeadm support and VM evidence after payload activation. |

Unsupported sysext surface:

- Package repository files or package-manager workflows.
- Shells or general scripting environments.
- Helm, Flux, Cilium, Rook, CoreDNS add-on management, or arbitrary Kubernetes
  distribution behavior.
- CNI policy beyond the minimal bootstrap fixture used by VM tests.

The Kubernetes sysext is selected from HTTPS payload bundles. Raw sysext paths
are build outputs and descriptor blobs, not user-facing activation input.

## Node Extension Payloads

Node extension bundles are the only supported way to add optional host-side
application payloads. Extension userland is not baked into the runtime rootfs.

Supported:

- HTTPS node extension bundle source/ref input.
- One app sysext payload per bundle.
- Katl-scoped systemd units, config handoff paths, status paths, capabilities,
  and health metadata declared by the bundle.

Unsupported:

- Arbitrary raw sysext activation.
- Arbitrary unit files or package names.
- Inline secrets.
- Ownership of kubeadm, Kubernetes add-ons, package managers, or non-Katl `/etc`
  paths unless a future app-specific decision explicitly takes that ownership.

## CI Artifact Boundaries

Artifact verification enforces the product boundary rather than a complete
Fedora file inventory.

The installer must contain Katl's installer executable, console, input handoff,
readiness service, and target. It must not contain VM-test support, package or
build tooling, Kubernetes payload tools, or a container runtime.

The runtime must contain Katl's agent, console, generation, boot-health, state-
projection, and persistent-access integration. It must not contain VM-test
support, package or build tooling, installer services, or Kubernetes payload
binaries that belong in the selected sysext. Its state and `/etc/kubernetes`
mount contracts remain exact because installed generations consume them.

The Kubernetes sysext must expose a coherent kubeadm, kubelet, kubectl, and
crictl payload compatible with the runtime interface. It must not absorb
package repositories or user-managed Kubernetes add-ons.

Exact Fedora paths, transitive utilities, kernel inventory, compression tuning,
and generated systemd text are implementation evidence, not supported surface.
The relevant VM journeys assert that the assembled artifacts actually boot,
install, provide access, activate services, and bootstrap Kubernetes. See
`testing-contract-policy.md`.
