# v0.1 Release Gate Matrix

Status: accepted gate plan for the v0.1 multi-node lifecycle proof.

This matrix defines what must be true before the v0.1 milestone can close. It
is intentionally command-shaped: later Beads may add gates, but they must name
the exact command they rely on and where its artifacts are retained.

## Host Tiers

Tier 0, unit and artifact host:

- Go toolchain available in the repository shell.
- `jq`, because VM runner setup and debug target summaries parse JSON with it.
- `scripts/mkosi` can run the containerized mkosi builder when artifact gates
  are selected.
- No libvirt, KVM, OVMF, network, or storage-pool access is required.

Tier 1, single-node VM host:

- Everything in Tier 0.
- `scripts/vmtest-run`, `virsh`, and an image tool such as `qemu-img` in `PATH`.
- Disk/image helper tools used by the VM harness: `mformat`, `mcopy`,
  `truncate`, `sfdisk`, `sha256sum`, `awk`, and `realpath`.
- Readable OVMF code and vars images, either discovered by the VM shell or set
  with `KATL_OVMF_CODE` and `KATL_OVMF_VARS`.
- Access to `qemu:///system`, or an explicit `KATL_VMTEST_LIBVIRT_URI`.
- Active libvirt network and storage pool, normally `default` or explicitly set
  with `KATL_VMTEST_LIBVIRT_NETWORK` and
  `KATL_VMTEST_LIBVIRT_STORAGE_POOL`.
- `/dev/kvm` when `KATL_VMTEST_KVM=on`.

Tier 2, multi-node Kubernetes VM host:

- Everything in Tier 1.
- Enough CPU, memory, disk, and libvirt lease capacity for three KatlOS nodes
  plus the operator/dev-host behavior modeled by the scenario.
- `/dev/vhost-vsock` when a scenario uses vsock.
- Kubernetes bootstrap host tools selected by the test fixture, including
  `kubectl` or `KATL_VMTEST_KUBECTL`.
- Network behavior sufficient for cross-node workload proof and future BGP
  peering to the dev host.

Host capability failures are not product failures. They must be recorded in the
VM world artifacts and mapped back to a host-capability Bead or to the Bead that
introduced the missing prerequisite.

## Retention And Debugging

All VM gates run through `scripts/vmtest-run`. The runner creates a world under
`${TMPDIR:-/tmp}/katl-vmtest/` and writes:

- `world.json`
- `host-capabilities.json`
- `run.json`
- `go-test.log`
- per-scenario `scenario.json` and `result.json`
- libvirt `domain.xml`, launch command, lease files, and serial logs
- operation records, kubeconfigs, diagnostic command records, and generated
  artifacts written by the selected scenario
- preserved live-domain debug summaries in `preserved-vm-debug.txt` when debug
  targets are retained

The default retention policy is `KATL_VMTEST_KEEP=failed`: passed run
directories are removed, while failed, setup-failed, and no-scenario-result run
directories are kept. Other supported policies are `always` and `never`.

The default debug preservation policy is `KATL_VMTEST_DEBUG_ON_FAILURE=auto`,
enabled for local runs and disabled in CI. Debug runs that need a live VM should
pass `--debug-on-failure` and a narrow `-run` pattern with `-failfast` so a
failure does not leave a fleet of domains running. Cleanup after live debugging
is explicit:

```sh
scripts/vmtest-debug /tmp/katl-vmtest/<run-id>
scripts/vmtest-clean /tmp/katl-vmtest/<run-id>
```

Long VM gates must pass an explicit Go test timeout. Use `-timeout`, not an
unbounded run. When debugging hangs, prefer one scenario and one count:

```sh
scripts/vmtest-run --debug-on-failure ./internal/vmtest/scenarios \
  -run '^TestInstalledRuntimeTwoNodeKubeadmJoinSmoke$' \
  -count=1 -failfast -timeout 45m
```

## Mandatory Gates

### Unit And Golden Gate

Command:

```sh
go test ./...
```

Tier: 0.

Required before: every code commit that closes a v0.1 Bead, unless the Bead
records a narrower justified gate and a follow-up to restore the full gate.

Expected artifacts: Go test output. This gate does not prove enabled VM, boot,
install, update, or kubeadm flows.

Failure mapping: failing package owner Bead, or a new regression Bead if the
failure is outside the active scope.

### Pull-Request VM Workflow

Command surface: `.github/workflows/vmtest.yml`.

Tier: 1 and selected Tier 2 capable-host gates.

The pull-request VM workflow runs for same-repository, non-draft pull requests
and manual dispatches on self-hosted capable runners labeled for `katl-vmtest`,
`libvirt`, `kvm`, `ovmf`, and `vsock`. It serializes the direct runtime,
first-install, installed-runtime ready/config-apply, and two-node bootstrap
gates through `nix develop .#vm` using the same `scripts/vmtest-run` commands
documented below, each with `-count=1`, `-failfast`, and an explicit Go test
`-timeout`.

Expected artifacts: the uploaded vmtest run directory for each matrix entry,
excluding large VM disk images, including `world.json`, `host-capabilities.json`,
`run.json`, `go-test.log`, scenario results, serial logs, VM metadata,
operation/generation metadata, disk metadata, and scenario summaries.

Failure mapping: failed setup or zero-scenario runs fail the workflow. Missing
declared runner capabilities are host-capability failures, not product skips.
Workflow artifacts should be used to file or update the owning Bead.

### Delivery Format Fixture Gate

Command:

```sh
go test ./internal/installer/sysextcatalog \
  ./internal/installer/kubernetesbundle \
  ./internal/installer/nodeextensionbundle -count=1
```

Tier: 0.

Required before: any milestone that consumes Kubernetes payload bundles or node
extension bundles.

Expected artifacts: in-test fixture roots for Kubernetes payload versions
`v1.36.0` and `v1.36.1`, node extension bundle fixtures, digest-addressed blob
checks, static HTTPS fetch tests, and rejection-path diagnostics.

Failure mapping: M0 delivery-format Beads for format regressions; consuming
milestone Beads for wrong assumptions about source/ref syntax.

### Artifact Build Gate

Command:

```sh
scripts/mkosi build-runtime
scripts/mkosi build-installer
scripts/mkosi build-katlos-install-image
```

Tier: 0 with mkosi/container builder access.

Required before: VM gates that depend on current runtime, installer, or KatlOS
install image artifacts.

Expected artifacts: mkosi output indexes, runtime root artifact, installer
artifact, KatlOS install image artifact, and resource-lock metadata under the
repo build output.

Failure mapping: artifact-production Beads for stale indexes, missing outputs,
or host-specific path leaks.

### Direct Runtime VM Gate

Command:

```sh
scripts/vmtest-run --artifact-set=runtime ./internal/vmtest \
  -run '^TestDirectRuntimeVMTestAgentSmoke$' \
  -count=1 -failfast -timeout 20m
```

Tier: 1.

Required before: runtime-root, userspace, agent, or boot-health changes can be
considered releasable.

Expected artifacts: direct-runtime scenario result, runtime serial log,
domain XML, launch command, VM agent health evidence, and host capabilities.

Failure mapping: runtime-root or VM harness Bead. Host capability gaps remain
host-capability Beads.

### First Install Gate

Command:

```sh
scripts/vmtest-run ./internal/vmtest \
  -run '^TestFirstInstallTargetDiskSerialSmoke$' \
  -count=1 -failfast -timeout 35m
```

Tier: 1.

Required before: disk layout, installer, boot handoff, generation-0, or install
artifact changes can be considered releasable.

Expected artifacts: installer serial log, target disk details, installed ESP
artifacts, installed runtime fixture publication, generation metadata, and
install status.

Failure mapping: installer state-machine, disk, boot, or artifact Bead depending
on the failed phase.

### Installed Runtime And Config Apply Gate

Command:

```sh
scripts/vmtest-run ./internal/vmtest \
  -run '^(TestInstalledRuntimeKubeadmReadySmoke|TestInstalledRuntimeKubeadmAPISmoke|TestInstalledRuntimeConfigApplyModesSmoke)$' \
  -count=1 -failfast -timeout 45m
```

Tier: 1.

Required before: `katlc` agent, generation metadata, config apply, kubeadm
projection, or installed-runtime fixture changes can be considered releasable.

Expected artifacts: installed runtime serial logs, operation records, generated
confext/generation files, kubeadm readiness evidence, and config-apply
diagnostic artifacts.

Failure mapping: generation/config apply/kubeadm projection Beads. Missing
fixture publication maps back to the first-install fixture Bead.

### Two-Node Kubernetes Bootstrap Gate

Command:

```sh
scripts/vmtest-run ./internal/vmtest/scenarios \
  -run '^TestInstalledRuntimeTwoNodeKubeadmJoinSmoke$' \
  -count=1 -failfast -timeout 60m
```

Tier: 2.

Required before: M4 Kubernetes bootstrap can close.

Expected artifacts: per-node operation records, operator kubeconfig and
metadata, kubeadm transcripts, node-local readiness/status snapshots,
cross-node workload artifacts, kubectl diagnostics, and Kubernetes pod/container
log summaries.

Failure mapping: kubeadm/bootstrap Beads for operation failures; VM harness Beads
for missing retained evidence; host-capability Beads for libvirt or network
capacity gaps.

Kube-proxy prerequisites are part of this gate. Failure diagnostics must capture
at least `conntrack`, `iptables-nft`, `ipvsadm`, `ipset`, loaded modules,
`modprobe -n -v ip_vs`, `modprobe -n -v br_netfilter`, and CNI fixture state.
If kube-proxy fails and these are absent, file or reopen a runtime artifact Bead
before debugging Kubernetes behavior.

### Three-Control-Plane Etcd Gate

Command:

```sh
scripts/vmtest-run ./internal/vmtest/scenarios \
  -run '^TestInstalledRuntimeThreeControlPlaneStackedEtcdSmoke$' \
  -count=1 -failfast -timeout 75m
```

Tier: 2.

Required before: M4 multi-control-plane lifecycle proof can close.

Expected artifacts: all two-node artifacts plus stacked-etcd verification,
control-plane inventory, etcd reports, and per-node bootstrap status.

Failure mapping: etcd ownership/bootstrap Beads for product failures; VM harness
Beads for missing evidence or fixture planning failures.

## Conditional Gates

### Operation-Backed Bootstrap Gate

Command:

```sh
scripts/vmtest-run ./internal/vmtest/scenarios \
  -run '^TestInstalledRuntimeTwoNodeOperationBackedBootstrapSmoke$' \
  -count=1 -failfast -timeout 60m
```

Tier: 2.

Mandatory when operation-backed bootstrap behavior changes. Deferred for Beads
that only touch artifact format or single-node runtime behavior.

### Sysupdate Gate

Command:

```sh
scripts/vmtest-run ./internal/vmtest \
  -run '^TestInstalledRuntimeSysupdateRootUKITransfer$' \
  -count=1 -failfast -timeout 45m
```

Tier: 1.

Mandatory when root update, UKI transfer, boot selection, rollback, or
systemd-sysupdate behavior changes. Deferred for pure Kubernetes/bootstrap
Beads until the v0.1 update milestone.

### Full Capable-Host Gate

Command:

```sh
nix develop .#vm -c env \
  KATL_VMTEST_RUN_ID=v0.1-capable-host-$(date -u +%Y%m%dT%H%M%SZ) \
  scripts/vmtest-run ./... -count=1 -failfast -timeout 90m
```

Tier: 2.

Mandatory before the final v0.1 release proof closes. It may be deferred from
individual implementation Beads when a narrower gate names the exact scenario
that covers the changed behavior.

## Deferred Until App-Specific Milestones

These are not optional for v0.1, but they do not block M1:

- BIRD extension bundle packaging and activation gate.
- BGP API VIP validation, rendering, and health gate.
- BGP peering from control planes to the dev host.
- Kubernetes upgrade from `v1.36.0` to `v1.36.1`.
- Config apply across a live multi-node Kubernetes cluster.
- Rollback or wipe/reinstall after a failed upgrade.

Each future Bead must add its exact command to this matrix or to a successor
gate document before closing.

## Failure-To-Bead Rule

When a gate fails:

1. Keep the run directory until the failure is classified.
2. Use `scripts/vmtest-debug` for preserved live domains and
   `scripts/vmtest-clean` after inspection.
3. Attach or reference the run directory, `go-test.log`, result JSON,
   serial logs, operation records, and relevant diagnostic files in the active
   Bead note.
4. If the failure is outside the active scope, create or reopen a concrete Bead
   and record the skipped gate on the active Bead.
5. Do not close a v0.1 Bead with a skipped mandatory gate unless the close reason
   names the host capability gap or follow-up Bead that owns the missing proof.
