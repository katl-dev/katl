# v0.1 Kernel And Firmware Support Policy

Status: accepted v0.1 support policy.

KatlOS v0.1 trims user-visible rootfs and installer tooling, not kernel
hardware breadth. The default policy is to ship the broad Fedora kernel module
set needed for normal bare-metal and VM installs, then verify representative
required classes in artifacts and VM gates. Kernel module pruning must be
explicit, narrow, and justified by this policy; it must not happen as a side
effect of userland surface minimization.

This policy is packaging and artifact support, not hardware certification. A
module, firmware blob, or successful VM boot proves that KatlOS includes the
expected software path. It does not prove every device in that hardware family
has been tested.

## Required In-Box Hardware Classes

The v0.1 installer and runtime images must keep in-box support for:

- EFI x86-64 systems booting systemd-boot and UKIs. The current executable VM
  proof is OVMF on KVM/libvirt.
- The installed KatlOS disk layout: GPT, vfat ESP and XBOOTLDR where present,
  SquashFS immutable root slots, ext4 state partitions, ext4 etcd partitions,
  and extra data disks using the schema-supported `ext4` or `xfs` filesystems.
- Common block storage controllers and virtual disks, including NVMe, AHCI/SATA,
  SCSI/SAS, USB storage, virtio-blk, and virtio-scsi.
- Common wired networking and VM networking, including virtio-net and normal
  Fedora in-tree Ethernet drivers. Katl configures networking with
  systemd-networkd; the kernel policy should not narrow hardware to the current
  VM NIC.
- Kubernetes host networking prerequisites: `br_netfilter`, `nf_conntrack`,
  `overlay`, `ip_vs`, `ip_vs_rr`, `ip_vs_wrr`, `ip_vs_sh`, nftables/iptables
  compatibility, IP forwarding, and the current CNI bridge path.
- VM and test-host primitives used by the supported harness, including virtio
  devices and the KVM/libvirt guest path.
- In-tree display and GPU kernel drivers for common non-proprietary hosts:
  `amdgpu`, Intel `i915`, and Intel `xe`, where Fedora packaging permits the
  driver and required redistributable firmware.

The installer and runtime must continue to include `kernel-modules`; replacing
it with only `kernel-modules-core` is not a v0.1 pruning strategy unless a later
decision names the exact retained hardware classes and gates the result.

## Firmware And Licensing

Firmware support is separate from module support. Current package locks include
`kernel-core`, `kernel-modules`, and `kernel-modules-core`, but they do not yet
prove any firmware or CPU microcode package is present. `katl-dty.16.2.7` must
turn this policy into artifact checks and either add the required redistributable
firmware packages or record a release-blocking exception.

The intended v0.1 policy is:

- Redistributable Fedora-packaged firmware needed by required in-box classes is
  allowed and should be included when the package license permits redistribution
  in KatlOS artifacts.
- GPU firmware needed by `amdgpu`, `i915`, or `xe` is expected out of box when
  Fedora ships it under redistributable terms.
- CPU microcode is allowed only through normal redistributable distro packages;
  if redistribution or artifact constraints are unclear, v0.1 must document the
  omission rather than silently imply coverage.
- Proprietary or out-of-tree driver stacks are not v0.1 in-box support.
  Examples include NVIDIA proprietary modules, Broadcom `wl`, DKMS-delivered
  drivers, vendor side-loaded kernel modules, and ZFS kernel modules.
- Highly specialized firmware or drivers can be future node extension or
  operator-managed territory only after a separate design defines trust,
  licensing, update, and rollback behavior.

If a required class cannot be supported because Fedora packaging splits firmware
in a way KatlOS cannot redistribute, the release notes and artifact checks must
say that directly. Absence must be explicit.

## Pruning Rules

Allowed v0.1 pruning:

- Unsupported userland tools, package managers, build tools, and generic admin
  conveniences named by the image-surface policy.
- Kernel debug symbols, documentation, source trees, build headers, and other
  non-runtime development payloads.
- A specific obsolete or specialized kernel module family only after a follow-up
  Bead records that the family is outside the required in-box classes and adds a
  negative artifact check or package-lock rule.

Disallowed v0.1 pruning:

- Common storage, filesystem, wired networking, virtualization, kube-proxy, or
  CNI-related modules needed by the classes above.
- `amdgpu`, Intel `i915`, or Intel `xe` merely because VM tests are headless.
- `ip_vs` or related kube-proxy modules while kube-proxy IPVS mode remains a
  supported or diagnosable bootstrap path.
- Any module or firmware removal justified only by the fact that the current VM
  smoke still boots.

The rootfs surface policy may deny stable operator entrypoints such as package
managers and compilers. It must not be used to infer that kernel hardware
support should be reduced to the VM fixture.

## Artifact Checks

The first executable checks should prove artifact contents and release
expectations without claiming hardware certification:

- Package-lock checks keep `kernel-core`, `kernel-modules`, and
  `kernel-modules-core` present for installer and runtime artifacts until a
  narrower module decision exists.
- Runtime-root checks verify Kubernetes prerequisite configuration and helpers:
  `nf_conntrack`, `br_netfilter`, IP forwarding, `conntrack`, `iptables-nft`,
  `ipset`, `ipvsadm`, nftables, and required CNI plugins.
- Module inventory checks inspect the built root or boot artifact and run
  equivalent `modprobe -n -v` probes for representative required modules:
  `br_netfilter`, `nf_conntrack`, `overlay`, `ip_vs`, `ip_vs_rr`, `ip_vs_wrr`,
  `ip_vs_sh`, `nvme`, `ahci`, representative `virtio_*` modules, `amdgpu`,
  `i915`, and `xe`.
- Firmware inventory checks report required firmware package or file coverage
  for the required GPU, network, storage, and CPU-microcode classes as present,
  absent-by-policy, or missing. A missing required firmware item is a release
  artifact failure, not a Kubernetes bootstrap debugging problem.
- UKI checks keep proving that the runtime boot asset contains a Linux payload,
  initrd, command line, and SquashFS root metadata.
- VM gates prove the current libvirt/KVM/OVMF boot and Kubernetes prerequisite
  behavior. The two-node Kubernetes bootstrap gate must capture loaded modules,
  `modprobe -n -v ip_vs`, `modprobe -n -v br_netfilter`, kube-proxy helper
  binaries, and CNI fixture state when kube-proxy fails.

Physical hardware coverage beyond these artifact checks is not implied by v0.1.
Adding certification language requires a separate test matrix, hardware lab
inventory, failure triage process, and release sign-off path.
